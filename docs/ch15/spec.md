# Team Lead 系统 Spec

## 背景
ch13 实现了子 Agent（独立上下文+受限工具），ch14 加了 Worktree 文件隔离。但当前架构是「一对一派发」——Lead 每派一个子 Agent 就得等它完成（或切后台），没有团队协作能力。多个子 Agent 之间零沟通，所有信息必须经 Lead 中转。

## 目标
- 把主 Agent 升级为 Team Lead，构建长期团队，并行派生多个队员
- 队员间通过共享任务列表和点对点消息直接协作
- Lead 支持 coordinator 模式——专注派人和决策，代码修改交给队员

## 功能需求
- **F1：** 抽象长期「小组」对象（名称/负责人/花名册/持久化位置），成员记角色/工作目录/运行后端/是否审批
- **F2：** 支持多种成员运行后端（独立终端窗格 / 同进程），按环境优先级自动选，不静默降级
- **F3：** 审批流程——需审批成员先发计划给 Lead，Lead 用结构化回复批准/驳回
- **F4：** 成员生命周期——完成自然停下标记空闲并通知 Lead，Lead 可从磁盘恢复上下文继续指派
- **F5：** 共享任务增删查改（带依赖字段），仅小组成员可见
- **F6：** 点对点消息——名称注册表+邮箱文件两段式，支持广播和结构化协议
- **F7：** 消息格式——发件人/正文/时间戳/已读/摘要，落盘自动补时间戳、默认未读
- **F8：** 邮箱并发安全——锁文件+重试+过期检测
- **F9：** Lead 发起流程——拆目标为带依赖任务→派生成员→git merge 合并→冲突解决/回滚上报
- **F10：** Coordinator 模式——双锁激活（config+环境变量），剥夺写文件工具，保留读+shell，只留派人/终止/消息/合并

## 非功能需求
- **N1：** 小组元数据在成员异常退出后不丢失
- **N2：** 锁文件超时 30s 可配
- **N3：** 消息读写不阻塞 Agent 主循环

## 不做的事
- 跨机器分布式团队
- 成员间实时流式通信
- 复杂任务依赖约束（条件分支/超时重试等）

## 验收标准
- **AC1：** Lead 创建小组 → 拆目标为 3 个任务写入共享清单 → 派生 2 个队员并行执行
- **AC2：** 队员 A 通过点对点消息把结果发给队员 B，B 收到后基于此继续工作
- **AC3：** 需审批队员发计划 → Lead 批准 → 队员执行；Lead 驳回 → 队员重新规划
- **AC4：** Coordinator 模式开启 → Lead 的 write_file 不可用，read_file/exec_command 可用
- **AC5：** 所有队员完成后 → Lead git merge → 无冲突自动合并 → 有冲突回报
- **AC6：** 队员完成任务后标记空闲 → Lead 发消息恢复上下文 → 继续工作不重新 spawn
- **AC7：** 两队员同时写同一邮箱 → 锁文件保护，后到者重试成功
