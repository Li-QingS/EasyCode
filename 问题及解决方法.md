# EasyCode 开发问题及解决方法

## ch03 工具系统

### 1. MarkdownRenderer 吞掉了 onToolCall（导致工具调用完全不显示）

**现象：** 用户问"读取项目结构"，SSE 日志显示模型返回了 `tool_use` 事件，但 TUI 没有任何反应，只有 `(1.8s)` 空计时。

**根因：** `MarkdownRenderer` 包装了 `StreamHandler`，但没有覆盖 `onToolCall()` 和 `onUsage()` 方法。Java 接口的 `default` 空实现在 `MarkdownRenderer` 层被调用，内层匿名类的 `onToolCall` 永远到不了。

**修复：** 在 `MarkdownRenderer` 中添加透传方法：
```java
@Override public void onToolCall(ToolCall call) { delegate.onToolCall(call); }
@Override public void onUsage(int in, int out) { delegate.onUsage(in, out); }
```

**教训：** 任何包装 `StreamHandler` 的中间层，必须把接口的**所有方法**都透传，哪怕当前没用到。default 方法不是"空"，是"阻断"。

---

### 2. `Config.systemPrompt` 反序列化后为 null

**现象：** 模型不知道有哪些工具可用，system prompt 实际发出去的是空字符串。

**根因：** `systemPrompt` 字段缺少 `@JsonProperty("system_prompt")` 注解。Jackson 反序列化 `easycode.yaml` 时，即使 YAML 中没有该字段，也可能因为 setter 的存在将其置为 null，覆盖 Java 字段的默认值。

**修复：**
1. 添加 `@JsonProperty("system_prompt")` 注解
2. 提供 `systemPromptSafe()` 方法，在值为 null/blank 时返回硬编码默认值
3. `AnthropicProvider.buildRequestBody()` 使用 `systemPromptSafe()` 而非 `systemPrompt()`

---

### 3. 嵌套 chatStream 导致 HTTP 连接死锁

**现象：** 工具执行成功后，二次 `chatStream` 的 spinner 永远不灭，SSE 日志为零。

**根因：** `handleToolCall` 在第一个流的 `onNext` 回调中直接调用 `startStreamingChat` → `chatStream` → `sendAsync().join()`。第一个 HTTP 连接未关闭，第二个 `sendAsync` 在同一个 `HttpClient` 上等待连接释放，形成事实死锁。

**修复：** 改为标记驱动——`handleToolCall` 只设 `needFollowUp = true` 后立即 return。外层 `startStreamingChat` 等第一个 `chatStream` 完全结束（连接关闭）后，检测标记再发起第二个 `chatStream`。

```java
private void startStreamingChat() {
    needFollowUp = false;
    doStreamingChat(conversation.getHistory());
    if (needFollowUp) {
        needFollowUp = false;
        doStreamingChat(conversation.getHistory());
    }
}
```

**教训：** 绝对不要在 HTTP 响应流的回调中发起新的 HTTP 请求（尤其是同一 HttpClient）。用标记 + 外部循环串行化。

---

### 4. 模型第一次请求的 `onComplete` 时机错乱

**现象：** 工具执行结果和二次请求之间出现一个孤立的 `(2.2s)` 计时行。

**根因：** 第一个 `chatStream` 的 `onComplete` 在 `needFollowUp` 流结束后才触发，但它不知道后续还有二次请求，照常打印了计时并尝试 `addAssistantMessage`。

**修复：** `onComplete` 中检查 `needFollowUp` 标志——如果为 true（即将发起二次请求），跳过计时行和 `addAssistantMessage`。

---

### 5. 模型反复多次调用工具

**现象：** 二次 `chatStream` 中模型拿到工具结果后，不是回答用户，而是又调了一个工具。

**根因：** system prompt 指令不够强——之前写的是"每次只调一个工具，拿到结果后再继续"，模型理解为可以继续调工具。加上 tools 数组让模型"看到"所有可用工具，激发了探索欲望。

**修复：**
1. system prompt 改为"严格只调一次工具。拿到工具结果后立即回答，绝对不要再调第二个工具"
2. TUI 层加 `needFollowUp` 深度保护——第二次 `chatStream` 中的工具调用不会再触发第三次

---

### 6. 请求体 body 过大导致 API 超时

**现象：** 工具结果（如 `ls -R` 输出）回灌进对话历史后，二次 `chatStream` 的请求体可能达到 5KB+，模型响应极慢或超时。

**修复：** `ExecCommandTool` 输出截断从 4000 字符改为 1000 字符。工具结果只需要让模型理解结构足够，不需要完整输出。

---

### 7. 对话后无输出（`onComplete` 清空了短回复）

**现象：** 简单问候"你好"后没有回复内容，只有计时。

**根因：** `onComplete` 中无条件调了 `stopSpinner()`，而 `stopSpinner` 打印的 `\r\033[K` 把同行的短回复清掉了（因为短回复最后一个 token 不含 `\n`）。

**修复：** 加 `spinnerStopped` 标记——`onToken`/`onToolCall` 中已经停过一次转圈，`onComplete` 再调时跳过清行。

---

### 8. Markdown 渲染裸漏 `##` 等符号

**现象：** 模型输出的 `## 标题`、`### 子标题` 等 Markdown 语法直接暴露在终端中。

**根因：** 原 `MarkdownRenderer` 只处理了 `**粗体**`，未处理标题语法。

**修复：** 增加行首 `#` 检测——连续 `#` 和紧随的空格被吞掉，标题文字自动加粗显示。

---

### 9. thinking 内容暴露给用户

**现象：** DeepSeek 模型始终返回 `thinking_delta`，用户看到模型的推理过程。

**根因：** 原 `parseSSELine` 把所有 `content_block_delta` 的 `text` 字段都转发给了 `onToken`，没区分 thinking 和 text。

**修复：** 只转发 `delta.type == "text_delta"` 的 token，`thinking_delta` 和 `signature_delta` 被过滤掉。

---

## 设计原则总结

1. **中间层透传原则：** 任何包装接口的类必须透传所有方法，default 空实现不是无害的
2. **HTTP 回调原则：** 不在响应流回调中发起新请求，用标记 + 外部循环串行化
3. **Jackson 默认值原则：** 有默认值的字段必须加 `@JsonProperty`，否则反序列化可能置 null
4. **输出行安全原则：** `\r` 和 ANSI 清屏序列可能破坏同行文本，结束时加 `\n` 保护
5. **Prompt 精确性原则：** system prompt 中的"可以"对模型意味着"不强制"，用"必须"、"严格"、"绝对不"
6. **工具结果精简原则：** 喂回模型的工具结果要截断，上下文窗口宝贵

---

### 10. read_file 缺乏行号和分段读取能力

**现象：** 模型读文件后无法精确引用行号，只能猜位置；读大文件一次性全返浪费 context window。

**修复：**
1. 每行加行号标注：`行号|内容`
2. 增加 `offset`（起始行）和 `limit`（最大行数）参数，默认 200 行
3. 末尾标注 `(文件共 N 行，已显示 L1-L2，用 offset=M 继续)`

---

### 11. read_file 无二进制/不可读文件检测

**现象：** 读到 `.class` 等二进制文件返回乱码，浪费 token 且误导模型。

**修复：** 双检测机制：
1. 扩展名黑名单（`.class`、`.jar`、`.png` 等 40+ 种）
2. 内容采样——前 512 字节中不可打印字符 > 30% 判为二进制
3. 判为二进制/不可读 → 直接返回错误，不读内容

---

### 12. 工具权限分级缺失

**现象：** 所有工具无差别执行，读写工具没有确认机制，安全性不足。

**修复：** `Tool` 接口加三个元数据方法：
```java
enum Permission { READ_ONLY, READ_WRITE }
default Permission permission() { return READ_WRITE; }
default boolean requiresApproval() { return false; }
enum State { ENABLED, DISABLED, APPROVAL_REQUIRED }
```
| 工具 | 权限 | 确认 |
|------|------|------|
| read_file / find_files / grep_code | 只读 | 否 |
| write_file / edit_file / exec_command | 读写 | 是 `[y/n]` |

---

### 13. 路径格式不兼容（反斜杠、Windows 盘符）

**现象：** 用户输入 `D:\agent project\EasyCode\src\...`，模型透传后工具报"文件不存在"。

**根因：** WSL 中 `D:\...` 不合法，需转为 `/mnt/d/...`；反斜杠在 Java Path 和 JSON 转义中都有问题。

**修复（三层）：**
1. **工具侧清洗：** `path.replace('\\', '/').trim()` + Windows 盘符正则转 WSL
2. **System prompt：** "所有文件路径使用正斜杠 /，优先使用相对路径"
3. **工具描述：** 明确写"支持相对/绝对路径，统一使用正斜杠"

```java
String path = input.get("path").asText().replace('\\', '/').trim();
if (path.matches("^[a-zA-Z]:/.*"))
    path = "/mnt/" + path.substring(0,1).toLowerCase() + path.substring(2);
```

---

### 14. exec_command 缺少退出码和 stderr

**现象：** `exec_command` 把 stderr 合并到 stdout，命令失败时看不出原因（如 `cat 不存在文件` 只返回空）。

**修复：**
1. 移除 `redirectErrorStream(true)`，分别读取 stdout 和 stderr
2. 结果末尾加 `[退出码: N]`，非 0 表示命令失败
3. stderr 输出标注 `[stderr]`，模型知道如何排查

---

## 设计原则补充

7. **路径鲁棒性原则：** 工具侧做路径清洗（反斜杠→正斜杠、Windows盘符→WSL），不依赖模型传正确格式
8. **输出可诊断原则：** 命令执行必须返回退出码和 stderr，模型才能判断成功/失败并自我纠正
9. **安全纵深原则：** 权限分级 + 终端确认 + 路径安全检查，三层不互信
10. **工具自描述原则：** 每个工具的 description() 必须包含：功能、该用/不该用场景、每个参数的含义、边界限制（至少 3-4 句）

---

### 15. needFollowUp 只允许 1 次跟进，多步任务被截断

**现象：** 模型调了 `exec_command ls` → `exec_command find` → 想调 `read_file` 但没有第三次机会，用户看到空白。

**根因：** `startStreamingChat` 只允许初始调用 + 1 次跟进。多步任务(探索→查找→读取→回答)需要 3-4 次。

**修复：** 跟进上限从 1 改为 3(for loop)，模型有足够回合完成完整工作流。

---

### 16. 模型持续用 exec_command 替代专用工具

**现象：** 模型总是用 `exec_command ls` / `exec_command find` 替代 `find_files`，工具描述区分不够强。

**根因：** 工具描述过于泛化，模型优先选自己熟悉的 Unix 命令。不是代码bug，是语义引导问题。

**修复：** 每个工具描述增加了与其他工具的区分语句（如"列出目录用 exec_command ls，搜索特定文件用 find_files"）。

---

### 17. 不停追加 system prompt 规则是反模式

**现象：** 每次模型调用工具出错都往 system prompt 加一条规则，导致 prompt 膨胀。

**根因：** 这是补丁式修复，没有解决根因。根因是模型缺少上下文或工具描述不够区分。

**修复：** 回退所有 prompt 规则，改为精简 system prompt + 强化工具描述本身。工具描述自带互斥指引，模型从工具定义就能判断选哪个。

---

### 18. ContextScanner 注入项目地图浪费 token

**现象：** 启动时把项目文件列表注入 system prompt，每次请求都重复传输。

**根因：** 设计失误——文件发现功能应该由 `find_files` 工具实现，不应该在 system prompt 里预存。

**修复：** 删除 ContextScanner 及 `projectContext` 字段，文件发现职责交还给 `find_files` 工具。

---

### 19. Tool 接口缺少分类元信息

**现象：** 工具没有分类，无法按类型管理或展示。

**修复：** 加 `Category` 枚举(SEARCH/FILE/SHELL) + `category()` 方法，ToolRegistry 按分类管理，/help 按分类展示。

---

### 20. exec_command 退出码语义不区分命令类型

**现象：** 所有命令非零退出码都标记为 error，但 grep exit1="没找到"不算错。

**修复：** 维护命令语义表 `EXIT_SEMANTICS`，对 grep/diff/find/cmp/rg 只有 >1 才算 error。加 `isDestructive()` 标记。

---


## ch09 命令系统

### 44. 命令注册与分发系统的设计动机

**背景:** 旧的 `/help` `/exit` `/perm` `/compact` `/resume` 硬编码在 Tui.start() 主循环里，if/else 链越来越长。新增命令需要改动 Tui 内部逻辑，命令和 UI 耦合紧密，无法做 Tab 补全。

**方案:** 设计了一套命令注册和分发机制：
1. `CommandRegistry` — 名称+别名映射，启动期冲突检测(冲突→抛异常终止)，支持补全候选生成
2. `CommandParser` — 斜杠前缀识别，`/name args` 解析，大小写不敏感
3. `CommandDef` — 命令元数据 + Builder 模式，处理函数为 `Function<String, CommandResult>`
4. `CommandResult` — 密封接口，6 种结果：Ok/Message/Prompt/NotFound/Error/Exit
5. `CommandDispatcher` — 分流器：`/` 开头走本地分发，否则作为对话消息
6. `UiController` 接口 — 抽象 UI 操作，命令实现不绑定 Tui

三类命令：LOCAL(纯本地) / UI(影响界面) / PROMPT(预设提示词注入对话)
内置 10 条命令：/help /compact /clear /plan /do /session /memory /permission /status /review

---

### 45. Tui.java 经过多次 Python 脚本修补导致结构损坏

**现象:** `mvn compile` 报 45 个错误，分别来自 BuiltinCommands.java 的"illegal line end in character literal"和 Tui.java 的"class, interface, enum, or record expected"。

**根因:** 使用 Python 脚本对 Tui.java 做了多次增量修改，每次脚本都在文件末尾附加代码块 + 额外闭合花括号，导致：
1. BuiltinCommands.java 中 `\n` 转义序列被 Python 字符串解析为真实换行，破坏了 Java 字符串
2. Tui.java 出现多次 `}` 闭合类，UiController 方法和 CommandCompleter 内部类被写到了类体之外

**修复:** 从 git 恢复原始 Tui.java 和 Main.java，然后用单次 Python 脚本生成完整的目标文件，不再做增量编辑。

**教训:** 复杂文件的修改使用一次性生成脚本优于多次增量编辑。每次增量都可能引入结构损坏，且难以回滚。

---

### 46. CommandResult switch 模式匹配不兼容 Java 17

**现象:** `mvn compile` 报错——sealed interface 的 switch 模式匹配需要 Java 21 预览特性。

**根因:** pom.xml 配置 `maven.compiler.source=17`，但 `switch (cr) { case CommandResult.Exit __ -> ... }` 使用了 Java 21 的模式匹配 + 未命名变量语法。

**修复:** 改为 `if (cr instanceof CommandResult.Exit)` 链式判断，兼容 Java 17。

**教训:** 使用语言特性前务必检查项目 target 版本。Java 17 支持 sealed interface + instanceof 模式匹配，但不支持 switch 穷举模式匹配。

---

### 47. Tui 与 Dispatcher 的循环依赖

**问题:** Tui 构造函数需要 `CommandDispatcher`，但 `CommandDispatcher.registerBuiltins()` 需要 `UiController`（即 Tui 实例）。

**方案:** `CommandDispatcher.setUi(UiController)` 延迟注入。装配顺序：
```
CommandRegistry → CommandDispatcher(reg, null) → Tui(..., dispatcher) → dispatcher.setUi(tui) → dispatcher.registerBuiltins()
```

---

### 48. CommandCompleter 类型冲突（自定义 Candidate vs JLine Candidate）

**问题:** `CommandRegistry` 定义了 `record Candidate(String name, String description)`，JLine 3 也有 `org.jline.reader.Candidate`。`CommandCompleter.complete()` 的参数 `List<Candidate>` 是 JLine 类型，但 `dispatcher.complete()` 返回 `List<CommandRegistry.Candidate>`。

**方案:** 在 `CommandCompleter` 中做类型转换——遍历 `matches` (CommandRegistry.Candidate)，逐一 `new Candidate(m.name(), ...)` 创建 JLine Candidate 添加到 `candidates` 列表。Java 编译器能根据 import 区分两个 Candidate 类型。

---

### 49. /permission /session 交互从纯文本改为数字菜单

**现象:** `/permission` 无参时只展示静态文本，用户不能切换，必须重新输入 `/permission plan`。

**根因:** 旧实现是纯文本输出，没有交互。用户期望像旧 `/perm` 一样能直接选择切换。

**修复:** `/permission` 无参时显示 `[1] DEFAULT` `[2] ACCEPT_EDITS` `[3] PLAN` `[4] BYPASS` 菜单，读取键盘数字 1-4 切换，Enter 取消。`/session` 同样改为列出可恢复会话并输入数字恢复。用 `System.in.read()` 直接读取（绕过 JLine 的 readline 模式）。

---

### 50. /help handler 和 printHelp() 重复

**现象:** Tui.java 中保留了 `printHelp()` 方法，但 `/help` 命令实际走 BuiltinCommands handler 生成文本并返回 Message。

**根因:** 重构时遗留的死代码——`printHelp()` 未被任何代码路径调用。

**修复:** 删除 `printHelp()` 方法，/help 完全走 BuiltinCommands → UiController.showMessage 路径。


## 设计原则再补充

11. **工具互斥原则：** 每个工具描述里必须写"不要用这个工具做什么，应该用哪个替代"，让模型在工具定义层面就能选对
12. **Prompt 最小化原则：** system prompt 只保留核心指令，工具选用逻辑应该编码在工具描述本身，而不是靠 prompt 规则堆叠
13. **多步串行原则：** 模型需要多步操作时，用标记驱动循环，每步 API 调用完全串行。上限设 3-5 轮防止无限循环
14. **元信息驱动原则：** 工具的权限、分类、破坏性等属性应该在接口层抽象，后续功能(权限系统、UI分组)直接复用

## ch04 Agent Loop

### 21. Spinner 线程与文本输出竞态导致文本丢失

**现象：** 模型流式返回文本时，内容与 spinner 的 `\r` 刷新字符交织，用户在终端看不到完整回复。

**根因：** spinner daemon 线程每 120ms 用 `\r` 原地刷新同一行，`stopSpinner` 用 `Thread.interrupt()` 无法立即终止——线程可能在 interrupt 后继续执行一次 `System.out.print("\r...")`，与 Tui 的 `\r\033[K` 清行序列和后续文本输出产生三向竞态。

**修复：**
1. 用 `volatile boolean spinnerRunning` 替代 interrupt——`stopSpinner` 先置 false 再清行
2. `stopSpinner` 中 `spinnerThread.join(200)` 等待线程真正退出后再清行
3. TextDelta 到达时若 spinner 仍在运行，先 `stopSpinner()` + `println()` 换行再输出文本

---

### 22. System prompt 被 sed 破坏导致模型不产出文本

**现象：** 修改 system prompt 后，模型对所有问题返回空文本，包括简单问候"你好"。

**根因：** `sed` 删掉 `"拿到结果再回答。"` 后，`"每次一个工具"` 直接接 `"路径统一用正斜杠"`，模型读到的指令变成强制性"每次都用工具"而缺了"之后要回答"，加上跨请求对话污染叠加，模型完全不产出文本。

**修复：** 恢复原 system prompt 结构 + 强化 `"优先凭自身知识回答；需准确内容时再使用工具"` + 恢复 `addUserMessage`/`trimToWindow` 调用。

---

### 23. 跨请求对话污染（用户消息不入历史 + trimToWindow 丢失）

**现象：** 第二个请求的 input_tokens 与第一个相同，模型看不到新问题。

**根因：** 重写 Tui 时删掉了 `conversation.addUserMessage(line)` 和 `trimToWindow()`——用户消息从未入历史，前一轮的重试提示残留在 conversation 中污染下一个请求。

**修复：** 恢复 `addUserMessage` + `trimToWindow` 调用，Tui 构造函数接收 `ConversationMgr` 和 `Config`。

### 24. 空文本检测逻辑缺陷（把无工具调用等同于完成）

## ch07 MCP 客户端

### 34. StdioTransport.send() 写入 json.toString() 而非 message.toString()

**现象:** 所有 MCP server 连接超时，stderr 显示 server 已启动 (`running on stdio`) 但 initialize 握手永远得不到响应，错误信息为 `null`。

**根因:** `StdioTransport.send()` 中写的是 `json.toString()`——`json` 是 `private static final ObjectMapper json` 实例，其 `toString()` 返回 `"com.fasterxml.jackson.databind.ObjectMapper@18f20ce8"`。server 收到无效 JSON 无法解析，自然不响应。而参数 `message` (实际的 JSON-RPC 消息) 被完全忽略。

**排查过程:**
1. 错误信息始终为 `null` → 改进错误处理，解包 `ExecutionException.getCause()`
2. 用 `echo '...' | node server.js` 手动测试 → server 正常响应，排除 server 端问题
3. 写 `TestStdioThreaded.java` 用 daemon 线程模拟 `StdioTransport` → 工作正常
4. 添加 debug 输出 `System.err.println("[mcp-debug] sending " + json.toString().substring(0, 80))` → 发现输出的是 ObjectMapper 哈希而非 JSON
5. `json` 变量名与"JSON 字符串"概念混淆，`json.toString()` 等价于 `ObjectMapper.toString()`

**修复:** `stdin.println(json.toString())` → `stdin.println(message.toString())`。同时清理了 `OutputStreamWriter` 未使用 import、daemon 线程 debug 输出、不必要的 `Thread.sleep()`。

**教训:**
- 变量命名极其重要——`json` 作为 ObjectMapper 实例名极易与 "JSON 字符串" 混淆
- 当代码行为与手动测试不一致时，添加 debug 输出追踪实际传输的数据
- `ExecutionException` 会包装超时异常，catch 时需解包 `e.getCause()`

### 35. ExecutionException 包装 TimeoutException 导致错误信息丢失

**现象:** `McpManager` 中 server 连接超时后，stderr 输出 `failed: null` 而非 `timed out (30s)`。

**根因:** `CompletableFuture.orTimeout(30, SECONDS)` 产生 `TimeoutException`，但 `future.get(30, SECONDS)` 会将其包装为 `ExecutionException`。`McpManager` 的 catch 块先 catch `TimeoutException` 再 catch `Exception`，导致 `ExecutionException` 被泛型 catch 捕获，`e.getMessage()` 为 null。

**修复:** 在 `catch (Exception e)` 中检查 `e.getCause() instanceof TimeoutException`，并增加 `e.getClass().getName()` 兜底输出。

### 36. McpManager.close() 缺少超时兜底

**现象:** `McpManager.close()` 顺序遍历所有 client 调用 `close()`，若某个 server 的 close 阻塞则整体退出卡死。

**修复:** 用 `CountDownLatch` + `CachedThreadPool` 并发关闭每个 client，`done.await(5, SECONDS)` 整体兜底 5 秒，超时后强制 `shutdownNow()`。

**修复:** 去掉 `final` 修饰符，改为 package-private class。同时为 `JsonRpcHandler` 添加 null transport 保护，避免测试中 `new McpClient(null, "test")` 抛 NPE。

## 设计原则再补充

20. **角色交替原则:** 任何修改对话历史的操作（摘要、恢复段拼接）必须保证 USER ↔ ASSISTANT 严格交替。拼接时检查首条消息角色，必要时插入占位消息。
21. **估算时序原则:** Token 估算器的 reset 必须在「取 after 值」之前执行，否则旧 anchor 会覆盖压缩后的实际值。
22. **变量命名警惕性:** `json` 作为 ObjectMapper 变量名是危险的——`json.toString()` 返回的是 ObjectMapper 实例信息，不是 JSON 字符串。用 `mapper` 或 `objectMapper` 更安全。
24. **命令去重原则:** 新系统引入新入口后，应清理旧入口。`/perm` 的 PLAN 选项覆盖了 `/plan` `/do` 的功能，旧命令应及时删除。
25. **不可覆盖编辑原则:** 修改复杂文件时，使用一次性生成脚本(从原始版本构建目标版本)优于多次增量编辑。增量编辑累积结构损坏的风险高。
26. **语言版本保守原则:** 使用语言新特性前必须检查项目 target 版本。不要假设、不要为了"好看"而启用预览特性。
27. **循环依赖打破原则:** 构造函数循环依赖时，用 setter 延迟注入("先创建，再连线")优于复杂的工厂或 DI 框架。简单且可测试。

### 44. Fork Skill 上下文溢出导致 MAX_ROUNDS（15轮）耗尽

**现象:** 用户运行 `/review`（fork 模式 Skill），后台线程跑满 15 轮后返回 `(Skill 达到最大轮次，未完成)`，审查任务未产出结果。

2. 每轮请求前调用 `ctxMgr.autoManage(forkConv, toolsJson)` 自动压缩
3. 追踪 token 用量（从 API usage 读取），超阈值触发压缩
4. 添加空输出重试 + nudge 消息机制（仿主循环 `EMPTY_TEXT_NUDGE`）

### 46. SkillRegistry 白名单死锁 — `whitelistActive` 永远为 false 🔴

**现象:** 白名单机制从第一天起就是死的——所有 Skill 激活后工具始终全量开放，`allowedTools` 限制形同虚设。

**根因:** `activate()` 方法通过 `activeToolWhitelist()` 来判断是否启用白名单，但 `activeToolWhitelist()` 第一行就是 `if (!whitelistActive) return Collections.emptySet()`。初始 `whitelistActive=false` → `activeToolWhitelist()` 返回空集 → `!空集.isEmpty()` = false → `whitelistActive` 被设为 false，形成循环自锁。

```java
// activate() 第 72 行
whitelistActive = !activeToolWhitelist().isEmpty();
//                 ↑
// activeToolWhitelist() 第 115 行:
//   if (!whitelistActive) return Collections.emptySet();
//   ↑ 先检查 whitelistActive（此时为 false），直接返回空集！
```

**修复:** 拆出 `computeToolWhitelist()` 方法——直接计算 activated Skill 的 allowedTools 并集，不检查 `whitelistActive` 标志。`activate()`/`deactivate()` 调用 `computeToolWhitelist()`，`activeToolWhitelist()` 先检查标志位再委托 `computeToolWhitelist()`。

**修改文件:** `SkillRegistry.java`。

**教训:** 自引用循环依赖是最隐蔽的初始化 bug。状态标志的计算逻辑不能依赖同一个标志本身。拆成纯计算函数 + 标志位守卫，永远不要在一个方法里同时「检查状态」和「计算决定该状态的值」。

### 47. `@JsonProperty("context_window")` 注解错挂到 `toolTimeout` 🔴

**现象:** YAML 配置 `context_window: 200000` 后，上下文窗口未生效，反而工具超时变成了 200 秒。这是一个**静默错误**——不抛异常，语义完全错误。

**根因:** `@JsonProperty("context_window")` 注解写在了 `toolTimeout` 字段上，而不是 `contextWindow` 字段。

```java
// 修复前
private Integer contextWindow;  // 无注解，Jackson 用 camelCase: "contextWindow"
@JsonProperty("context_window")
private int toolTimeout = 30;   // ← 注解在这里！
```

**修复:** 将 `@JsonProperty("context_window")` 移到 `contextWindow` 字段。

**修改文件:** `Config.java`。

**教训:** JSON 反序列化注解的错位是零编译错误的语义 bug。唯一防御手段是：写完配置类必写一个单元测试，用 YAML 字符串反序列化后断言每个字段值。

---

### 48. `SkillRegistry.deactivate()` 未注销 Skill 专属工具

**现象:** 目录型 Skill（含 `tool.json` 定义专属工具）激活后再停用，专属工具残留在 `ToolRegistry` 中。再次激活同名工具时静默覆盖，旧实例未清理。

**根因:** `activate()` 注册 `SkillProvidedTool` 到全局 `ToolRegistry`，但 `deactivate()` 没有任何移除逻辑。`ToolRegistry` 也没有 `remove()` 方法。

**修复:**
1. `ToolRegistry` 添加 `remove(String name)` 方法
2. `SkillRegistry` 添加 `Map<String, Set<String>> activatedTools` 字段追踪每个 Skill 注册的工具名
3. `activate()` 记录到 `activatedTools`，`deactivate()` 遍历移除
4. `clearActivated()` 同步清理

**修改文件:** `ToolRegistry.java`、`SkillRegistry.java`。

**教训:** 有注册必有注销。任何「全局注册表 + 临时生命周期」的组合，必须在停用时恢复注册表原状。对称性不是可选项，漏掉一边就是资源泄漏。

---

### 49. `SkillProvidedTool` 脚本路径未做 Shell 转义

**现象:** 当 Skill 的 `tool.json` 中 `script` 路径包含空格（如 `/home/user/my skills/run.sh`），Shell 解析为多个参数，命令执行失败或产生注入风险。

**根因:** `SkillProvidedTool.execute()` 中 `def.scriptPath().toString()` 直接拼接到 Shell 命令字符串，未做转义：

```java
// 修复前
String cmd = def.scriptPath().toString() + " " + shellEscape(jsonInput);
// scriptPath 未转义，路径中的空格被 Shell 当分隔符
```

**修复:** 对脚本路径也做 Shell 转义：
```java
String cmd = shellEscape(def.scriptPath().toString()) + " " + shellEscape(jsonInput);
```

**修改文件:** `SkillRegistry.java`（`SkillProvidedTool.execute()` 内）。

**教训:** 任何外部输入拼接到 Shell 命令前必须转义。两条数据路径→两次转义，不能只转义一半。

### 50. `LoadSkillTool` 中 `NullNode` 绕过空值检查

**现象:** JSON 输入 `{"name": null}` 时，`input.get("name").asText()` 返回字符串 `"null"`，通过 `isBlank()` 检查，最终调用 `registry.load("null")` 导致 `IllegalArgumentException`。

**根因:** Jackson 的 `NullNode.asText()` 返回字符串 `"null"` 而非 Java `null`。原代码的 `name == null` 检查永远不触发。

**修复:** 先取 `JsonNode`，判 `isNull()`，再取文本：
```java
JsonNode nameNode = input.get("name");
if (nameNode == null || nameNode.isNull()) return error("缺少 name 参数");
String name = nameNode.asText();
if (name.isBlank()) return error("name 参数不能为空");
```

**修改文件:** `LoadSkillTool.java`。

**教训:** `JsonNode.asText()` 对 `NullNode` 返回 `"null"`（4 个字符的字符串），不是 Java `null`。JSON 库的语义陷阱——永远先判 `isNull()`，再取值。

### 51. `SkillWatcher` 目录硬编码相对路径 + 不检测文件内容变更

**现象:**
1. 项目 Skill 目录使用 `Path.of(".easycode/skills")` 相对路径，依赖 JVM 工作目录。如果程序启动后 cwd 变化，热更新监控错误目录
2. `dirLastModified()` 取的是目录 mtime，Linux 上目录 mtime 只在增删文件时更新，**修改 Skill 文件内容不触发热更新**

**修复:**
1. 构造函数接受 `projectRoot` 参数：`projectRoot.resolve(".easycode/skills")`
2. `dirLastModified()` 改为递归遍历目录下所有文件，取最大 mtime
3. `Main.java` 传入 `Path.of("").toAbsolutePath()`

**修改文件:** `SkillWatcher.java`、`Main.java`。

**教训:** 相对路径是隐式依赖，构造函数应接受绝对基础路径。文件热更新必须检测文件内容变更，不能依赖目录 mtime（此行为跨平台不一致，Linux 上尤其坑）。

### 52. `SkillExecutor.resolveProvider()` 遗漏 Config 字段

**现象:** Skill 指定不同 model 时，fork 循环使用的 `Config` 缺少 `contextWindow` 和 `toolTimeout`，`ContextManager` 获得不准确的上下文窗口值，可能提前或延迟触发压缩。

**根因:** `resolveProvider()` 只拷贝了 `protocol/model/baseUrl/apiKey` 四个字段。

**修复:** 补上 `fc.setContextWindow(config.contextWindow())` 和 `fc.setToolTimeout(config.toolTimeout())`。

**修改文件:** `SkillExecutor.java`。

**教训:** 对象拷贝必须显式列出所有字段，漏一个就是静默 bug。优先用 `BeanUtils.copyProperties` 或 builder 模式减少遗漏，或写测试断言所有字段值一致。

---

### 53. `SkillLoader.validateAllowedTools()` 用 try-catch 做控制流

**现象:** 校验 allowedTools 合法性时用 `try { get() } catch (IllegalArgumentException)` 代替正常判断。

**根因:** `ToolRegistry` 已有 `getQuiet()` 方法返回 null，应该直接用。

**修复:** `if (toolRegistry.getQuiet(toolName) == null) throw ...`

**修改文件:** `SkillLoader.java`。

**教训:** try-catch 应该保留给真正异常的路径。用已有 API 的正常返回值做判断，性能和可读性都更好。

## 设计原则再补充

28. **注册对称性原则:** 任何资源的 `register()` 必须配 `deregister()`/`remove()`。生命周期不对称 = 资源泄漏。
29. **Shell 转义全覆盖原则:** 所有拼接到 Shell 命令的外部输入都需要转义，一个都不能漏。
30. **NullNode 陷阱原则:** `JsonNode.asText()` 对 `NullNode` 返回字符串 `"null"`，必须 `isNull()` 前置判断。
31. **相对路径显式化原则:** 项目内所有文件路径应以构造函数注入的 `projectRoot` 为基准 resolve，不依赖 JVM cwd。
32. **目录 mtime 不可靠原则:** 文件热更新必须检测文件级 mtime，不能依赖目录 mtime（跨平台行为不一致）。
33. **Config 拷贝全覆盖原则:** 构造派生 Config 时必须显式拷贝所有字段，漏一个即静默 bug。
