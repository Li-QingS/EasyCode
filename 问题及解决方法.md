# EasyCode 开发问题及解决方法

## ch03 工具系统

### 1. MarkdownRenderer 吞掉了 onToolCall（导致工具调用完全不显示）

**现象：** 用户问"读取项目结构"，SSE 日志显示模型返回了 `tool_use` 事件，但 TUI 没有任何反应，只有 `(1.8s)` 空计时。

**根因：** `MarkdownRenderer` 包装了 `StreamHandler`，但没有覆盖 `onToolCall()` 和 `onUsage()` 方法。Java 接口的 `default` 空实现在 `MarkdownRenderer` 层被调用，内层匿名类的 `onToolCall` 永远到不了。

**修复：** 在 `MarkdownRenderer` 中添加透传方法：
```java
@Override public void onToolCall(ToolCall call) { delegate.onToolCall(call); }
@Override public void onUsage(int in, int out) { delegate.onUsage(in, out); }
```

**教训：** 任何包装 `StreamHandler` 的中间层，必须把接口的**所有方法**都透传，哪怕当前没用到。default 方法不是"空"，是"阻断"。

---

### 2. `Config.systemPrompt` 反序列化后为 null

**现象：** 模型不知道有哪些工具可用，system prompt 实际发出去的是空字符串。

**根因：** `systemPrompt` 字段缺少 `@JsonProperty("system_prompt")` 注解。Jackson 反序列化 `easycode.yaml` 时，即使 YAML 中没有该字段，也可能因为 setter 的存在将其置为 null，覆盖 Java 字段的默认值。

**修复：**
1. 添加 `@JsonProperty("system_prompt")` 注解
2. 提供 `systemPromptSafe()` 方法，在值为 null/blank 时返回硬编码默认值
3. `AnthropicProvider.buildRequestBody()` 使用 `systemPromptSafe()` 而非 `systemPrompt()`

---

### 3. 嵌套 chatStream 导致 HTTP 连接死锁

**现象：** 工具执行成功后，二次 `chatStream` 的 spinner 永远不灭，SSE 日志为零。

**根因：** `handleToolCall` 在第一个流的 `onNext` 回调中直接调用 `startStreamingChat` → `chatStream` → `sendAsync().join()`。第一个 HTTP 连接未关闭，第二个 `sendAsync` 在同一个 `HttpClient` 上等待连接释放，形成事实死锁。

**修复：** 改为标记驱动——`handleToolCall` 只设 `needFollowUp = true` 后立即 return。外层 `startStreamingChat` 等第一个 `chatStream` 完全结束（连接关闭）后，检测标记再发起第二个 `chatStream`。

```java
private void startStreamingChat() {
    needFollowUp = false;
    doStreamingChat(conversation.getHistory());
    if (needFollowUp) {
        needFollowUp = false;
        doStreamingChat(conversation.getHistory());
    }
}
```

**教训：** 绝对不要在 HTTP 响应流的回调中发起新的 HTTP 请求（尤其是同一 HttpClient）。用标记 + 外部循环串行化。

---

### 4. 模型第一次请求的 `onComplete` 时机错乱

**现象：** 工具执行结果和二次请求之间出现一个孤立的 `(2.2s)` 计时行。

**根因：** 第一个 `chatStream` 的 `onComplete` 在 `needFollowUp` 流结束后才触发，但它不知道后续还有二次请求，照常打印了计时并尝试 `addAssistantMessage`。

**修复：** `onComplete` 中检查 `needFollowUp` 标志——如果为 true（即将发起二次请求），跳过计时行和 `addAssistantMessage`。

---

### 5. 模型反复多次调用工具

**现象：** 二次 `chatStream` 中模型拿到工具结果后，不是回答用户，而是又调了一个工具。

**根因：** system prompt 指令不够强——之前写的是"每次只调一个工具，拿到结果后再继续"，模型理解为可以继续调工具。加上 tools 数组让模型"看到"所有可用工具，激发了探索欲望。

**修复：**
1. system prompt 改为"严格只调一次工具。拿到工具结果后立即回答，绝对不要再调第二个工具"
2. TUI 层加 `needFollowUp` 深度保护——第二次 `chatStream` 中的工具调用不会再触发第三次

---

### 6. 请求体 body 过大导致 API 超时

**现象：** 工具结果（如 `ls -R` 输出）回灌进对话历史后，二次 `chatStream` 的请求体可能达到 5KB+，模型响应极慢或超时。

**修复：** `ExecCommandTool` 输出截断从 4000 字符改为 1000 字符。工具结果只需要让模型理解结构足够，不需要完整输出。

---

### 7. 对话后无输出（`onComplete` 清空了短回复）

**现象：** 简单问候"你好"后没有回复内容，只有计时。

**根因：** `onComplete` 中无条件调了 `stopSpinner()`，而 `stopSpinner` 打印的 `\r\033[K` 把同行的短回复清掉了（因为短回复最后一个 token 不含 `\n`）。

**修复：** 加 `spinnerStopped` 标记——`onToken`/`onToolCall` 中已经停过一次转圈，`onComplete` 再调时跳过清行。

---

### 8. Markdown 渲染裸漏 `##` 等符号

**现象：** 模型输出的 `## 标题`、`### 子标题` 等 Markdown 语法直接暴露在终端中。

**根因：** 原 `MarkdownRenderer` 只处理了 `**粗体**`，未处理标题语法。

**修复：** 增加行首 `#` 检测——连续 `#` 和紧随的空格被吞掉，标题文字自动加粗显示。

---

### 9. thinking 内容暴露给用户

**现象：** DeepSeek 模型始终返回 `thinking_delta`，用户看到模型的推理过程。

**根因：** 原 `parseSSELine` 把所有 `content_block_delta` 的 `text` 字段都转发给了 `onToken`，没区分 thinking 和 text。

**修复：** 只转发 `delta.type == "text_delta"` 的 token，`thinking_delta` 和 `signature_delta` 被过滤掉。

---

## 设计原则总结

1. **中间层透传原则：** 任何包装接口的类必须透传所有方法，default 空实现不是无害的
2. **HTTP 回调原则：** 不在响应流回调中发起新请求，用标记 + 外部循环串行化
3. **Jackson 默认值原则：** 有默认值的字段必须加 `@JsonProperty`，否则反序列化可能置 null
4. **输出行安全原则：** `\r` 和 ANSI 清屏序列可能破坏同行文本，结束时加 `\n` 保护
5. **Prompt 精确性原则：** system prompt 中的"可以"对模型意味着"不强制"，用"必须"、"严格"、"绝对不"
6. **工具结果精简原则：** 喂回模型的工具结果要截断，上下文窗口宝贵

---

### 10. read_file 缺乏行号和分段读取能力

**现象：** 模型读文件后无法精确引用行号，只能猜位置；读大文件一次性全返浪费 context window。

**修复：**
1. 每行加行号标注：`行号|内容`
2. 增加 `offset`（起始行）和 `limit`（最大行数）参数，默认 200 行
3. 末尾标注 `(文件共 N 行，已显示 L1-L2，用 offset=M 继续)`

---

### 11. read_file 无二进制/不可读文件检测

**现象：** 读到 `.class` 等二进制文件返回乱码，浪费 token 且误导模型。

**修复：** 双检测机制：
1. 扩展名黑名单（`.class`、`.jar`、`.png` 等 40+ 种）
2. 内容采样——前 512 字节中不可打印字符 > 30% 判为二进制
3. 判为二进制/不可读 → 直接返回错误，不读内容

---

### 12. 工具权限分级缺失

**现象：** 所有工具无差别执行，读写工具没有确认机制，安全性不足。

**修复：** `Tool` 接口加三个元数据方法：
```java
enum Permission { READ_ONLY, READ_WRITE }
default Permission permission() { return READ_WRITE; }
default boolean requiresApproval() { return false; }
enum State { ENABLED, DISABLED, APPROVAL_REQUIRED }
```
| 工具 | 权限 | 确认 |
|------|------|------|
| read_file / find_files / grep_code | 只读 | 否 |
| write_file / edit_file / exec_command | 读写 | 是 `[y/n]` |

---

### 13. 路径格式不兼容（反斜杠、Windows 盘符）

**现象：** 用户输入 `D:\agent project\EasyCode\src\...`，模型透传后工具报"文件不存在"。

**根因：** WSL 中 `D:\...` 不合法，需转为 `/mnt/d/...`；反斜杠在 Java Path 和 JSON 转义中都有问题。

**修复（三层）：**
1. **工具侧清洗：** `path.replace('\\', '/').trim()` + Windows 盘符正则转 WSL
2. **System prompt：** "所有文件路径使用正斜杠 /，优先使用相对路径"
3. **工具描述：** 明确写"支持相对/绝对路径，统一使用正斜杠"

```java
String path = input.get("path").asText().replace('\\', '/').trim();
if (path.matches("^[a-zA-Z]:/.*"))
    path = "/mnt/" + path.substring(0,1).toLowerCase() + path.substring(2);
```

---

### 14. exec_command 缺少退出码和 stderr

**现象：** `exec_command` 把 stderr 合并到 stdout，命令失败时看不出原因（如 `cat 不存在文件` 只返回空）。

**修复：**
1. 移除 `redirectErrorStream(true)`，分别读取 stdout 和 stderr
2. 结果末尾加 `[退出码: N]`，非 0 表示命令失败
3. stderr 输出标注 `[stderr]`，模型知道如何排查

---

## 设计原则补充

7. **路径鲁棒性原则：** 工具侧做路径清洗（反斜杠→正斜杠、Windows盘符→WSL），不依赖模型传正确格式
8. **输出可诊断原则：** 命令执行必须返回退出码和 stderr，模型才能判断成功/失败并自我纠正
9. **安全纵深原则：** 权限分级 + 终端确认 + 路径安全检查，三层不互信
10. **工具自描述原则：** 每个工具的 description() 必须包含：功能、该用/不该用场景、每个参数的含义、边界限制（至少 3-4 句）
