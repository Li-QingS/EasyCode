# MewCode

我正在构建一个终端 AI 编程助手（类似 Claude Code），项目名叫 EasyCode，使用 [Java] 实现。

## 语言
中文回答，中文注释。

## 测试

开发完功能后，用 tmux 做端到端测试：

1. 在 tmux 中启动 EasyCode
2. 输入一段真实的对话请求
3. 观察 EasyCode 是否正确调用工具、生成回复
4. 对照 checklist.md 逐项验收

## 工具描述规范

每个工具的 `description()` 是给模型看的"使用说明书"，必须极其详细，包含以下四部分：

### 1. 工具功能
一句话说明这个工具做什么。例如："读取项目中的文件内容并返回"。

### 2. 使用场景
明确列出**应该用**和**绝对不应该用**的场景。例如：
- 应该用：需要查看 pom.xml 内容、阅读 Java 源码
- 不应该用：查看项目结构（用 find_files 或 exec_command）、修改文件（用 edit_file）

### 3. 参数说明
逐一说明每个参数的含义、类型、是否必填，以及传不同值的行为差异。例如：
```
参数 path（string，必填）：相对于项目根目录的文件路径。如 "pom.xml" 表示根目录的 pom.xml，"src/main/java/com/easycode/Main.java" 表示嵌套路径。
```

### 4. 注意事项与限制
列出边界情况、限制、可能出错的点。例如：
- 文件超过 10KB 会被截断
- 不返回二进制文件内容
- 路径不存在时返回错误而不是空字符串

### 编写原则
- 每个工具描述至少 3-4 句，复杂工具可以更多
- 用中文写，因为 system prompt 是中文
- 从模型的角度思考：模型看了这个描述后，能不能准确判断何时该调用这个工具、传什么参数、如何处理失败
- 同一行信息不要重复说——system prompt 已经说了"拿到结果后立即回答"，工具描述里不用再说





## 调试原则

程序运行出现 bug 时，必须先分析根因，从原理上排查并修改逻辑设计，杜绝使用"兜底机制""fallback 补打"等方式绕过问题。每一个异常行为背后都有确定的代码缺陷，找到它、修复它，而不是在上面再盖一层。
